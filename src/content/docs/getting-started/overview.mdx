## Before you begin

- You do not need to know the full architecture yet.
- You only need to understand the basic flow from modeling to execution to reports.
- If you already installed the app and want to move faster, you can skip to [Quick Start](/getting-started/quick_start/).

## What you'll finish with

<OutcomeCard title="A usable mental model" tone="success">
  You should leave this page knowing what AppraiseJS stores, what it generates,
  how it runs tests, and where results show up when something passes or fails.
</OutcomeCard>

## Step-by-step procedure

1. Start with the authored assets: modules, suites, cases, template steps, locators, tags, and environments.
2. Understand that AppraiseJS turns those authored assets into executable artifacts and keeps them aligned through sync.
3. See test runs as the operational layer where you choose environment, browser, worker count, and scope.
4. Treat reports, logs, traces, and dashboard metrics as the inspection layer that closes the feedback loop.

## Checkpoint

- You can explain the difference between authored assets, generated artifacts, runs, and reports.
- You know that `Settings -> Sync` is the recovery point when UI data and generated files drift apart.
- You know the next practical page is [Installation](/getting-started/installation/).

## Common mistakes

- Treating AppraiseJS like a pure reporting tool instead of a full authoring and execution workflow.
- Assuming runs can succeed before environments, locators, and test cases exist.
- Jumping into advanced architecture pages before you have a first successful run.
